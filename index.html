Apollo-1 Foundation Model #

Apollo-1 is the first neuro-symbolic foundation model, purpose-built for task-oriented agents. It unifies generation and control in a single architecture, delivering both conversational flexibility and strict behavioral certainty.

Architecture

The Apollo-1 architecture consists of three core components:

Component Function
Domain-Agnostic Encoder Translates natural language into symbolic state
Stateful Reasoning Loop Maintains symbolic state, computes actions, creates executable plans
Domain-Agnostic Decoder Generates natural language responses from symbolic output
Key Innovation

Perception is probabilistic, but given the same state, the Reasoner always makes the same decision. This delivers behavioral certainty for enterprise applications.

Two Agent Types

Apollo-1 distinguishes between two fundamentally different agent paradigms:

Agent Type Priority Use Cases
Open-ended Agents Flexibility for individual users Coding assistants, personal AI, creative tools
Task-oriented Agents Organizational policy enforcement Airline booking, banking, insurance, support

Apollo-1 specifically targets task-oriented use cases where behavioral certainty is non-negotiable. These agents must enforce organizational policies absolutely while maintaining natural conversational flow.

Key Advantages

Composability

Truths defined once apply across all relevant contexts automatically.

Maintainability

Single-source updates propagate universally across the system.

Auditability

Every decision traces to specific firing constraints for compliance.

Separation of Concerns

Operators define requirements without implementation details.

The Truths Paradigm #

Rather than scripting step-by-step workflows or relying on probabilistic LLM decisions, Apollo-1 uses a "Truths" paradigm. Operators declare domain truths, and the model interprets user input, computes explicit symbolic state, evaluates applicable constraints, then executes deterministically.

How It Works

The model interprets user input → computes explicit symbolic state → evaluates applicable constraints → executes deterministically. Once state is correctly interpreted, compliance becomes deterministic rather than probabilistic.

Constraint Types

Type Purpose Example
Policy Constraints Business rules requiring enforcement "Block disputes after 8 days from transaction"
Confirmation Constraints Actions requiring explicit user consent "Confirm before processing payment"
Authentication Constraints Actions requiring identity verification "Verify ID for refunds over $200"

Unlike soft instructions or prompt engineering, constraints enforce at the architectural level. They are not suggestions that the LLM might ignore — they are hard rules that the symbolic reasoning engine enforces deterministically.

Conflict Resolution

When multiple constraints apply to the same entity or action, stricter actions take precedence:

Priority Order (Highest to Lowest) 
BLOCK  >  ASK_A_QUESTION  >  ACTIVATE_ANOTHER_TOOL  >  TEXT

This ensures that blocking constraints always win over less restrictive actions, maintaining safety and compliance.

Agents #

Agents are the top-level configuration that defines your conversational AI. Each agent has an objective, guardrails, tone of voice, and associated tools.

agent.aui.json 
{
  "$schema": "https://aui.dev/schema/aui.schema.json",
  "agent": {
    "code": "retail-support-agent",
    "version": "1.0.0",
    "description": "Customer support agent for retail banking",
    "language": "en",

    "objective": "Help retail banking customers with account inquiries, transaction disputes, and general support questions.",

    "guardrails": "Never share customer account details without identity verification. Always verify identity before processing financial transactions. Escalate to human agent for disputes over $5000.",

    "tone_of_voice": {
      "guidelines": "Professional, empathetic, and helpful",
      "tone": "customer-focused",
      "brevity": "concise"
    },

    "scope": {
      "org_id": "bank-corp",
      "account_id": "retail-division"
    }
  }
}

Agent Properties

Property Type Description
code string Unique identifier for the agent
version string Semantic version (e.g., 1.0.0)
objective string Primary purpose of the agent
guardrails string Safety guidelines and restrictions
tone_of_voice object Communication style settings
context object Static or API-based context data

Parameters #

Parameters are typed attributes that can be extracted from user input, populated from API responses, or used in both directions. They form the foundation of your agent's data model.

parameters.aui.json 
{
  "$schema": "https://aui.dev/schema/aui.schema.json",
  "parameters": [
    {
      "code": "transaction-id",
      "type": "string",
      "usage": "output",
      "description": "Unique identifier for a transaction"
    },
    {
      "code": "dispute-reason",
      "type": "enum",
      "usage": "input",
      "description": "Reason for the dispute",
      "values": ["unauthorized", "duplicate_charge", "product_damaged"]
    },
    {
      "code": "transaction-amount",
      "type": "number",
      "usage": "output",
      "description": "Dollar amount of the transaction"
    }
  ]
}

Parameter Types

Type Description
string Text values
number Numeric values (integers or decimals)
boolean True/false values
enum Predefined set of allowed values
date Date values (YYYY-MM-DD format)
array List of values

Usage Types

Usage Description
input Extracted from user messages
output Populated from API responses
all Can be both extracted and populated

Type-Specific Operators

Different parameter types support different comparison operators in rule conditions:

Type Supported Operators
string equals, not_equals, contains, not_contains, is_empty, is_not_empty, about (semantic)
number equals, not_equals, greater_than, less_than, greater_or_equal, less_or_equal, between
enum equals, not_equals, is_empty, is_not_empty
date equals, greater_than (after), less_than (before), between, more_than_x_days, less_than_x_days, exactly_x_days
boolean equals (true/false)
array contains, not_contains, is_empty, is_not_empty, any_have_value, all_have_value
Semantic Matching with "about"

The about operator uses AI-powered semantic matching, allowing conditions like "user_question about 'refund'" to match variations like "I want my money back" or "Can I get a refund?"

Entities #

Entities group related parameters into logical units. They represent the "nouns" of your agent's domain — objects like Products, Transactions, or Users.

entities.aui.json 
{
  "$schema": "https://aui.dev/schema/aui.schema.json",
  "entities": [
    {
      "code": "Transactions",
      "description": "Customer transaction records",
      "params": [
        "transaction-id",
        "transaction-amount",
        "transaction-status"
      ]
    },
    {
      "code": "UserProfile",
      "description": "Customer profile information",
      "params": [
        "customer-id",
        "account-tier"
      ]
    }
  ]
}
Best Practice

Entities are automatically populated from API responses through integration response_mapping. Design your entities to mirror your domain model for seamless data flow.

Tools #

Tools are the actionable capabilities of your agent. Each tool encapsulates logic to perform a specific function, with defined triggers, required parameters, integrations, and rules.

tools/create-dispute.aui.json 
{
  "$schema": "https://aui.dev/schema/aui.schema.json",
  "tool": {
    "code": "CREATE_DISPUTE",
    "objective": "When the user wants to dispute a transaction or report fraudulent activity",
    "status": true,
    "main_tool": true,
    "requires_confirmation": true,
    "response_type": "text",
    "tone_of_voice": "empathetic",

    "config": {
      "triggers": {
        "when": {
          "any": [
            { "method": "extracted_params_in_current_turn", "params": ["dispute-reason"] }
          ]
        }
      },
      "params": {
        "required": ["transaction-id", "dispute-reason"],
        "optional": ["supporting-documentation"]
      }
    },

    "integrations": [
      { "ref": "lookup-transaction" },
      { "ref": "create-dispute-api" }
    ],

    "rules": {
      "post": {
        "success": [
          {
            "when": {},
            "then": {
              "action": "TEXT",
              "message": "Dispute created successfully. Case #{{Disputes.dispute-id}}"
            }
          }
        ],
        "fail": [
          {
            "when": {},
            "then": {
              "action": "ACTIVATE_ANOTHER_TOOL",
              "tool": "ESCALATE_TO_AGENT"
            }
          }
        ]
      }
    }
  }
}

Tool Execution Flow

When a tool is invoked, Apollo-1 processes it through a prioritized logic sequence:

┌─────────────────────────────────────────────────────────────────┐ │ TOOL EXECUTION FLOW │ └─────────────────────────────────────────────────────────────────┘ User Message │ ▼ ┌─────────────────┐ │ 1. CONSTRAINTS │ ◀── Are there blocking conditions? (highest priority) │ Check │ └────────┬────────┘ │ ┌─────┴─────┐ │ │ BLOCK PASS │ │ ▼ ▼ ┌──────┐ ┌─────────────────┐ │ END │ │ 2. MISSING INFO │ ◀── Is required information missing? └──────┘ │ Rules │ └────────┬────────┘ │ ┌─────┴─────┐ │ │ MISSING FILLED │ │ ▼ ▼ ┌───────────┐ ┌─────────────────┐ │ ASK/BLOCK │ │ 3. INTEGRATIONS │ ◀── Execute API calls └───────────┘ └────────┬────────┘ │ ┌─────┴─────┐ │ │ SUCCESS FAIL │ │ ▼ ▼ ┌───────────┐ ┌───────────┐ │ 4. POST │ │ 4. POST │ │ SUCCESS │ │ FAIL │ │ Rules │ │ Rules │ └───────────┘ └───────────┘
Execution Order Matters

Constraints are always evaluated first. Blocking behaviors take precedence over information gathering, which takes precedence over execution. This ensures safety rules cannot be bypassed.

Tool Activation

Tools can be activated through multiple mechanisms:

Activation Method Description
Intent Classification Model matches user intent to tool objective (primary mechanism)
Structured Triggers Parameter extraction or conversation events trigger the tool
Rule Activation Another tool's rule activates this tool via ACTIVATE_ANOTHER_TOOL
Continuation Tool continues from previous turn (awaiting params/confirmation)

Tool States

State When It Occurs Actions
Missing Info Required parameters are missing Ask user, activate another tool, call API, or block
Success Integration returned valid data Apply situational rules based on returned data
Fail API error or no results Activate fallback tool, try another API, expand search

Tool Properties

Property Type Description
code string Unique identifier (SCREAMING_SNAKE_CASE)
objective string When to activate this tool
main_tool boolean If true, agent can activate directly
requires_confirmation boolean Ask user confirmation before executing
response_type enum text or text_and_widget
brevity enum concise, normal, or verbose

            "code": "Disputes",
            "identifier": "id",
            "paths": ["*"],
            "mappings": [
              { "key": "id", "param": "dispute-id" },
              { "key": "status", "param": "dispute-status" }

Rules & Constraints #

Rules define the behavior of your agent. Constraints are hard rules that must be enforced, while guardrails are soft guidance.

Rule Types

Type Use For Example
Policy Constraints Business rules "Block disputes for transactions older than 8 days"
Confirmation Constraints User consent "Require confirmation before processing payment"
Authentication Constraints Identity verification "Require ID verification for refunds over $200"

Constraint Example

rules.aui.json 
{
  "rules": {
    "pre": {
      "constraint": [
        {
          "code": "block-banned-users",
          "description": "Block all actions for banned users",
          "applies_to": [],  // Empty = ALL tools
          "when": {
            "all": [
              {
                "method": "user_profile",
                "param": "is-banned",
                "operator": "equals",
                "value": true
              }
            ]
          },
          "then": {
            "action": "BLOCK",
            "message": "Your account has been suspended."
          }
        }
      ]
    }
  }
}

Condition Methods

Conditions evaluate various aspects of the conversation state:

Method Description Example
param_exists Check if a parameter has a value { "method": "param_exists", "param": "transaction-id" }
param_missing Check if a parameter is missing { "method": "param_missing", "param": "dispute-reason" }
param_value Compare accumulated parameter value { "method": "param_value", "param": "amount", "operator": "greater_than", "value": 1000 }
param_extracted_current Check parameter extracted in current turn { "method": "param_extracted_current", "param": "product-type", "operator": "about", "value": "clothing" }
entity_param Check entity parameter value { "method": "entity_param", "entity": "Product", "param": "price", "operator": "less_than", "value": 100 }
entity_param_referenced Check if entity was referenced in current turn { "method": "entity_param_referenced", "entity": "CreditCards" }
user_profile Check user profile attribute { "method": "user_profile", "param": "account-tier", "operator": "equals", "value": "premium" }
user_question Semantic match on user message { "method": "user_question", "operator": "about", "value": "refund" }
conversation_started True at the start of conversation { "method": "conversation_started" }
current_url Check URL context (web widget) { "method": "current_url", "operator": "contains", "value": "/checkout" }
tool_required_params_filled All required params are filled { "method": "tool_required_params_filled" }
tool_optional_params_filled All optional params are filled { "method": "tool_optional_params_filled" }

Operators

Operator Description
equals / not_equals Exact match comparison
contains / not_contains String or array contains
greater_than / less_than Numeric comparison
between Value within range
about Semantic/AI-evaluated match
more_than_x_days / less_than_x_days Date comparison

Rule Actions

Action Description Additional Properties
TEXT Display a message to the user message (supports {{Entity.param}} interpolation)
ASK_A_QUESTION Ask user for information message, params (parameters to collect)
BLOCK Block the action and end tool execution message (reason for blocking)
BLOCK_AND_ACTIVATE_ANOTHER_TOOL Block current tool and activate fallback tool, message
ACTIVATE_ANOTHER_TOOL Hand off to another tool tool (tool code to activate)
ADD_REQUIRED_PARAM Dynamically require parameters params, reason
RETRY Retry the current operation max_retries
CALL_API Trigger an integration call integration (integration code)

Rule Placement

Rules are organized by when they execute relative to the tool's integration calls:

Location Category Purpose
Global pre.constraint Apply to all tools, block/redirect before execution
Tool Pre-Execution pre.constraint Tool-specific blocking conditions
pre.missing_info Handle missing parameter scenarios
Tool Post-Execution post.success Handle successful integration response
post.fail Handle failed integration response
Conditional Required Parameters

Use config.params.conditional in tools to dynamically add required parameters based on conditions. This is useful for cases like "require documentation for disputes over $1000".

Response Schema #

Field Type Description

timestamp datetime Interaction timestamp
duration_ms number Processing time in milliseconds
input object User's message and metadata
context object State from previous interactions
understanding object How input was interpreted
decisions array Sequential chain of decisions
response object Final response to user

Response Types

Type Description

question Asking user for information
confirmation_request Asking user to confirm an action

Type Description

Configuration Options

Option Type Default Description
networkApiKey string - Required authentication key
environment ApolloEnvironment Gcp Gcp or Azure
timeoutInSeconds number 60 Request timeout
maxRetries number 2 Max retry attempts
REST API Usage 
// Create a new task
const taskResponse = await client.controllerApi.createTask({
  user_id: 'user123',
  task_origin_type: 'web-widget'
});

// Send a message
const messageResponse = await client.controllerApi.sendMessage({
  task_id: taskResponse.id,
  text: 'I need help with my order',
  is_external_api: true,
  context: { url: 'https://example.com' }
});

// Get task messages
const messages = await client.controllerApi.getTaskMessages(taskResponse.id);

// List user tasks
const tasks = await client.controllerApi.listUserTasks({
  user_id: 'user123',
  page: 1,
  size: 10
});
WebSocket Usage 
// Connect to WebSocket for streaming responses
const socket = await client.apolloWsSession.connect({
  debug: false,
  reconnectAttempts: 3,
  headers: { 'x-network-api-key': 'YOUR_NETWORK_API_KEY' }
});

socket.on('open', () => {
  socket.sendUserMessage({
    task_id: 'task-id',
    text: 'Hello!'
  });
});

socket.on('message', (message) => {
  if (message.text) {
    console.log('Response:', message.text);
  }
});

socket.on('error', (error) => console.error('Error:', error));
socket.on('close', () => console.log('Disconnected'));

Schema Reference: Agent #

Complete reference for the agent configuration schema.

Property Type Required Description
code string Yes Unique agent identifier
version string Yes Semantic version (e.g., 1.0.0)
description string No What this agent does
language string No Primary language code (default: en)
scope object No Deployment scope (org_id, account_id, network_id)
objective string No Primary purpose of the agent
guardrails string No Safety guidelines and restrictions
tone_of_voice object No Communication style settings
context object No Static or API-based context

Schema Reference: Tool #

Property Type Required Description
code string Yes Unique tool identifier (SCREAMING_SNAKE_CASE)
objective string Yes When this tool should be activated
rules object Yes Pre and post execution rules
status boolean No Whether tool is enabled (default: true)
main_tool boolean No Can agent activate directly (default: true)
requires_confirmation boolean No Ask confirmation before executing (default: false)
response_type enum No text | text_and_widget
tone_of_voice string No Override agent tone for this tool
brevity enum No concise | normal | verbose
config object No Triggers and parameters configuration
integrations array No Integration references to execute

Schema Reference: Parameter #

Property Type Required Description
code string Yes Unique identifier (kebab-case)
type enum Yes string | number | enum | date | array | boolean
usage enum Yes input | output | all
description string Yes What this parameter represents
values string[] No Allowed values for enum type
nested_params string[] No Child parameter codes
question string No Question to ask when collecting

Schema Reference: Integration #

Property Type Required Description
code string Yes Unique integration identifier
type enum Yes REST | MCP | RAG | INTERNAL_CONTEXT | STRUCTURED | GEN_AI
description string No What this integration does
endpoint object No URL, method, auth, headers
request_schema array No Request field definitions
test_curl string No cURL command for testing
response_mapping object No Maps response to entities/params

Schema Reference: Rule #

Property Type Required Description
code string No Unique rule identifier
description string No What this rule does
applies_to string[] No Tool codes (empty = all tools)
when object Yes Condition group (all/any/not)
then object Yes Action to take

          { "method": "user_profile", "param": "tier", "operator": "equals", "value": "basic" }
        ]
      }
    ]
  },
  "then": {
    "action": "ADD_REQUIRED_PARAM",
    "params": ["verification-document"]
  }
}
